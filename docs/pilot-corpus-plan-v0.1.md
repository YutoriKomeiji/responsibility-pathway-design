# RPD Pilot Corpus Plan v0.1

## Objective

The first RPD case corpus should test whether the case definition, coding manual, evidence tiers, disagreement process, and negative-case rules are usable before any broad empirical claim is attempted.

The pilot is a methods test, not a validation study.

## Proposed pilot composition

A minimum pilot may contain eight to twelve bounded cases distributed across at least four case families.

A preferred initial sequence is:

1. one synthetic calibration case derived from the existing AI-assisted benefit review example;
2. two public documentary breakdown cases;
3. two repair cases with visible before/after process evidence;
4. one null or rival-success case;
5. one negative case in which intervention burden or delay increased;
6. one inconclusive case with material evidence gaps;
7. optional boundary cases from a different sector or organizational form.

The final pilot composition should be determined by evidence access and governance constraints, not by a target outcome.

## Pilot questions

The pilot should answer:

- Can researchers bound a pathway case consistently?
- Can different coders locate the same transitions?
- Which code families generate repeated disagreement?
- Are unknown and not-applicable states used correctly?
- Do evidence tiers distinguish documentary, exercise, and operational support?
- Can negative and null cases be retained without reinterpretation as success?
- Does the scorecard add information beyond narrative analysis?
- Which fields create disproportionate collection burden?
- Can confidentiality be preserved without removing analytically necessary context?
- Which propositions are not observable with available evidence?

## Work packages

### WP1 — Protocol freeze

Freeze versions of:

- case corpus protocol;
- coding and adjudication manual;
- empirical study record;
- observation log;
- comparative scorecard;
- result report;
- corpus manifest.

Record all deviations rather than editing the protocol silently during coding.

### WP2 — Case discovery and screening

For every candidate case, record:

- discovery source;
- preliminary pathway boundary;
- case family;
- likely evidence tier;
- inclusion and exclusion checks;
- confidentiality status;
- expected affected-party evidence;
- reason for admission, exclusion, or quarantine.

### WP3 — Calibration

Use at least two cases for coder training:

- one relatively clear case;
- one ambiguous or incomplete case.

Calibration should identify codebook defects before formal agreement measurement. Calibration results should not be mixed with the final agreement estimate unless declared.

### WP4 — Independent coding

Each eligible case should be coded independently by at least two coders where feasible.

Coders should not see:

- the other coder's assignments;
- the expected theory result;
- a preferred design conclusion;
- adjudicated results from later cases that could bias earlier coding.

### WP5 — Agreement and disagreement analysis

Report both agreement statistics and qualitative disagreement classes.

Priority outputs:

- code family agreement;
- transition-boundary agreement;
- evidence-tier agreement;
- confidence distribution;
- unknown and missing rates;
- adjudication rate;
- codebook-ambiguity count;
- theory-boundary disagreements.

### WP6 — Cross-case analysis

Cross-case synthesis should begin with configurations and mechanisms rather than a global score.

Compare:

- pathway breakdown location;
- intervention timing;
- authority–capability alignment;
- evidence continuity;
- contestability and recovery;
- burden distribution;
- pattern composition;
- observed defeaters;
- contextual conditions;
- negative, null, and rival explanations.

### WP7 — Method revision

At pilot close, issue one of the following for each protocol component:

- retain;
- clarify wording;
- split a code or field;
- merge redundant codes;
- restrict scope;
- add missing context;
- suspend use;
- retire.

Changes should be linked to observed pilot evidence.

## Stop conditions

Pause or stop the pilot when:

- confidentiality cannot be maintained;
- case admission would create material risk to affected parties;
- permission conditions are unclear;
- coders cannot establish a bounded pathway after adjudication;
- source provenance cannot be reconstructed;
- conflicts of interest are undisclosed or unmanaged;
- the study begins treating documentation quality as pathway effectiveness;
- the corpus becomes systematically selected for RPD-supportive outcomes;
- an external publication or deployment decision is being inferred without human authorization.

## Deliverables

The pilot should produce:

1. a versioned corpus manifest;
2. admitted, excluded, quarantined, corrected, and withdrawn case counts;
3. coded case packets;
4. an agreement and disagreement report;
5. negative, null, and rival-success findings;
6. a protocol usability report;
7. a theory and codebook revision ledger;
8. a recommendation on whether a larger corpus is justified.

## Success criteria

The pilot succeeds when it reveals where the method works and where it fails. It does not require favorable RPD results.

Minimum indicators of a useful pilot include:

- transparent case selection;
- preserved source provenance;
- visible coder disagreement;
- at least one serious negative, null, rival, or inconclusive analysis;
- explicit protocol revisions;
- bounded claims;
- a reproducible path from source material to case-level findings.

## Human gate

Collection involving confidential records, interviews, affected parties, external organizations, public release, or research publication requires explicit human authorization and any applicable ethics, legal, contractual, or institutional review.
